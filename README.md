# SheetSearch AI

**A natural language query engine for structured tabular data.**

Upload a CSV or XLSX file. Ask questions in plain English. Get precise, deterministic answers — backed by an execution trace.

> No SQL. No hardcoded schemas. No LangChain. No vector DB.

## Key Features

- **Deterministic execution** — query results are computed by Python functions, not generated by an LLM
- **Multi-step query plans** — filter → group → sort → limit pipelines, expressed as structured JSON
- **Hallucination prevention** — all filters are derived from user-extracted constraints; the LLM cannot invent filters
- **Semantic validation** — the generated plan is validated against extracted constraints before execution; invalid steps are stripped or repaired
- **Automatic schema inference** — column types (boolean, numeric, date, string) detected on upload; works with any spreadsheet
- **Self-repair loop** — if the plan fails validation, the agent repairs it and retries up to 3 times
- **Full execution trace** — every step exposes rows-in / rows-out and millisecond latency
- **Dataset-aware suggestions** — the chat UI generates contextual question prompts from the uploaded schema

## How It Works

A user question goes through six stages before results are returned:

### 1. Constraint Extraction
Deterministic rules parse the question for numeric thresholds, date ranges, boolean flags, categorical values, sort direction, and limit counts.

```
"Top 3 office locations by total salary among non-remote employees hired after 2017"

→ boolean:   remote = false
→ date:      hire_date > 2017-01-01
→ aggregate: SUM(salary) GROUP BY office_location
→ sort:      DESC
→ limit:     3
```

### 2. Query Planning (LLM)
GPT-4o-mini receives the question, the schema, and the extracted constraints. It returns a structured JSON execution plan:

```json
{
  "steps": [
    { "tool": "filter",          "args": { "column": "remote",          "op": "eq",  "value": false } },
    { "tool": "filter",          "args": { "column": "hire_date",       "op": "gt",  "value": "2017-12-31" } },
    { "tool": "group_aggregate", "args": { "group_by": "office_location", "op": "sum", "column": "salary" } },
    { "tool": "sort",            "args": { "column": "salary",          "ascending": false } },
    { "tool": "limit",           "args": { "n": 3 } }
  ]
}
```

### 3. Semantic Validation
Every filter step in the plan is checked against the extracted constraints. Steps that were not grounded in user intent are flagged as violations and stripped.

### 4. Plan Repair Loop
If the plan is invalid or incomplete (missing expected filters, wrong column), the agent repairs it and retries. Up to 3 repair attempts before falling back.

### 5. Deterministic Execution
Each plan step maps to a pure Python function operating on a Pandas DataFrame. The LLM has no involvement in result computation.

### 6. Structured Response
The API returns the result, the answer type (`aggregate`, `group_aggregate`, `rows`), a confidence score, and the full execution trace.

## Example Query

**Input:**
```
Top 3 office locations by total salary among non-remote employees hired after 2017
```

**Execution Plan:**
```
Step 1  filter          remote = false              500 → 312 rows   2ms
Step 2  filter          hire_date > 2017-12-31      312 → 198 rows   1ms
Step 3  group_aggregate SUM(salary) by office_loc   198 → 8 groups   3ms
Step 4  sort            salary DESC                 8 → 8 rows       0ms
Step 5  limit           n=3                         8 → 3 rows       0ms
```

**Response:**
```json
{
  "answer_type": "group_aggregate",
  "groups": [
    { "group": "New York",   "value": 1842000, "count": 67 },
    { "group": "London",     "value": 1614000, "count": 61 },
    { "group": "Singapore",  "value": 1203000, "count": 70 }
  ],
  "confidence": 0.94,
  "ms_total": 312
}
```

## Supported Query Types

| Category | Example |
|---|---|
| Numeric filter | `"Employees with salary above 120,000"` |

## Schema Inference

Column types are inferred automatically on upload. The rules are applied per-column over all non-null values:

| Inferred Type | Rule |
|---|---|
| `boolean` | ≥ 90% of values are `true / false / yes / no / 1 / 0` |
| `numeric` | ≥ 80% of values parse as a number |
| `date` | ≥ 80% of values match a known date format |
| `string` | Everything else |

Type information drives constraint extraction, filter validation, and frontend schema display.

## Running Locally

### Prerequisites

- Python 3.13+
- Node.js 18+
- OpenAI API key

### 1. Clone & install backend

```bash
git clone <repo-url>
cd sheetsearchai

# Recommended: uv
uv sync

# Alternative: pip
pip install -r requirements.txt
```

### 2. Configure environment

```bash
cp .env.example .env
```

Edit `.env`:

```env
OPENAI_API_KEY=sk-...

# Optional: Google Sheets read source
GOOGLE_SERVICE_ACCOUNT_FILE=sheetsearchkey.json
SHEET_ID=your_google_sheet_id
```

### 3. Start the backend

```bash
uv run uvicorn app.main:app --port 8001 --reload
```

API available at `http://localhost:8001`.

### 4. Start the frontend

```bash
cd frontend2
npm install
npm run dev
```

UI available at `http://localhost:5173`.

## API Reference

### `POST /upload`

Upload a CSV or XLSX file.

```json
// Response
{
  "ok": true,
  "dataset_id": "abc123",
  "columns": ["name", "salary", "department"],
  "types": { "salary": "numeric", "department": "string" },
  "row_count": 500
}
```

### `POST /agent/ask`

Run a natural-language query against an uploaded dataset.

```json
// Request
{
  "question": "Top 3 office locations by average salary",
  "dataset_id": "abc123"
}

// Response
{
  "ok": true,
  "answer_type": "group_aggregate",
  "groups": [{ "group": "NYC", "value": 112000, "count": 45 }],
  "confidence": 0.92,
  "trace": [
    { "tool": "group_aggregate", "rows_before": 500, "rows_after": 8, "ms": 4 },
    { "tool": "sort",            "rows_before": 8,   "rows_after": 8, "ms": 0 },
    { "tool": "limit",           "rows_before": 8,   "rows_after": 3, "ms": 0 }
  ],
  "plan": { "steps": [...] }
}
```

### `GET /api/stats?dataset_id=abc123`

Returns schema, column names, sample rows, and row count.

## Stress Testing

A 50-query stress test is included to verify correctness across all query categories against a deterministic 500-row synthetic dataset.

```bash
uv run python final_50_stress_test.py
```

**Expected pass rate: 50/50 (100%)**

Categories covered: basic numeric, boolean, date, group aggregation, sort + limit, HAVING, multi-step, combined constraints, and edge cases.
